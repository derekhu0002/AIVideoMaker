# AI Video Maker 演示手册

本文件面向**演示人员**，用于照着讲、照着操作，不要求理解代码实现细节。

如果你只需要知道怎么演示，请重点看：
- [演示前准备](#1-演示前准备)
- [启动方式](#2-启动方式)
- [推荐演示路径](#3-推荐演示路径)
- [失败时如何 fallback 演示](#7-如果现场某一步失败怎么兜底)

---

## 1. 演示前准备

建议在演示前确认以下内容：

1. 当前机器已安装 Python 3
2. 当前仓库可正常执行：

```bash
python3 run.py
```

3. 浏览器可访问：

```text
http://127.0.0.1:8000
```

4. 如果要演示 OpenCode sandbox 多 Agent，建议提前确认本机当前环境能正常运行 sandbox bridge
5. 演示时建议使用一个全新项目，避免历史数据干扰讲解

建议提前准备一组容易讲述的项目参数，例如：
- 项目名称：本地 AI 视频演示
- 主题：如何用 AI 提升短视频创作效率
- 受众：个人创作者
- 风格：清晰、干货
- 时长：30 秒

---

## 2. 启动方式

在仓库根目录执行：

```bash
python3 run.py
```

启动后打开浏览器访问：

```text
http://127.0.0.1:8000
```

进入后你会看到：
- 左侧：项目创建与项目列表
- 右侧：工作台主区域

## 3. 推荐演示路径

这是最推荐的一条完整演示路径，可以从头讲到尾。

### 第一步：创建项目

操作：
1. 在左侧填写项目名称、主题、受众、风格、时长
2. 点击“创建本地项目”

建议讲解：
- 这是一个**本地部署、本机浏览器访问**的 AI 视频制作网站
- 所有项目状态、产物、日志、确认记录都落在本地目录，便于追溯和审计

---

### 第二步：执行桥接驱动生成

操作：
1. 选中刚创建的项目
2. 点击“**桥接驱动生成**”

建议讲解：
- 这里不是只跑本地 stub，而是会先触发一次真实的 **OpenCode sandbox bridge**
- bridge 的作用是：在隔离的 runtime 沙箱里做一次多 Agent 团队验证，再把结果回写到当前项目
- 如果 bridge 成功，系统会继续推进主流程生成
- 如果 bridge 不可用，系统仍可 fallback 到本地 stub，保证演示不断

此时建议展示：
- “最近沙箱桥接”区域
- workflow 区域里的最近运行记录

---

### 第三步：查看阶段化产物

操作：
1. 向下查看脚本、分镜、配音文案、字幕、封面、发布素材建议

建议讲解：
- 当前系统采用**阶段化产物模型**
- 每个阶段都可以独立查看、编辑、确认、重生
- 用户不是被一次性自动生成绑死，而是可以人工参与控制

此时建议重点指出：
- 当前版本号
- 当前状态
- 最近 Agent
- 是否已冻结
- 是否因上游变化变成“待复核”

---

### 第四步：人工确认阶段产物

操作：
1. 对每个阶段点击“确认当前版本”

建议讲解：
- 这里体现的是“**人工确认关口**”
- 只有确认过的内容，才会进入下一步导出和发布准备
- 这保证了用户对内容质量和发布边界的控制权

---

### 第五步：演示冻结 / 待复核 / 解冻重生

推荐操作：
1. 在某个已确认阶段上展示“已冻结”状态
2. 如需要，可以点击“解冻并重生”演示受控修改

建议讲解：
- 已确认阶段会被冻结
- 如果上游变了，下游会被标记为待复核
- 这符合真实创作协作场景，而不是无约束覆盖结果

如果时间有限，这一步可以口头说明，不一定必须操作。

---

### 第六步：生成成片导出记录

操作：
1. 点击“生成成片导出记录”

建议讲解：
- 当前不是完整视频渲染引擎，而是**manifest 驱动的本地导出闭环**
- 它会生成一组可展示、可审计、可交接的本地产物

此时建议展示：
- 成片导出记录区域
- 导出产物路径

---

### 第七步：生成发布素材包

操作：
1. 点击“准备发布素材包”

建议讲解：
- 发布素材包包含标题、简介、标签建议、人工检查清单
- 它是“发布准备材料”，不是自动发布动作

---

### 第八步：生成人工发布交接

操作：
1. 在“人工确认发布交接”区域点击“记录人工发布交接”

建议讲解：
- 这是整个演示的关键边界：
  - 系统只生成交接材料
  - 最终发布仍由人手动完成
- 当前版本**不支持无确认自动直发**

此时建议展示：
- 发布状态
- handoff 记录
- checklist HTML 路径
- fallback actions

---

### 第九步：演示本机安全动作

推荐操作：
1. 点击“打开导出目录”
2. 点击“打开交接页”
3. 点击“记录复制发布文案”

建议讲解：
- 这些动作都是**本机安全动作**
- 系统记录了 launch evidence，方便后续审计
- 即使打开动作失败，也会记录 fallback 证据，不会影响整体演示闭环

---

## 4. 演示时建议重点讲的亮点

### 亮点 1：本地部署、本机浏览器访问
推荐讲法：

> 这个系统不是云端网页，而是直接运行在用户自己的电脑上，通过本机浏览器访问。这样更容易控制隐私、素材和操作边界。

### 亮点 2：多 Agent 团队不是纯概念，而是有真实 sandbox bridge
推荐讲法：

> 系统不是简单写死一个生成函数，而是能在仓库里的独立 OpenCode sandbox 中跑一次真实的多 Agent runtime，再把桥接结果回写到主流程里。

### 亮点 3：创作过程是可编辑、可确认、可重生的
推荐讲法：

> 这里不是一次性自动生成结果，而是把脚本、分镜、字幕、封面等都做成阶段化产物，允许人工修改、确认、重生，并保留版本和审计记录。

### 亮点 4：可追溯、可审计
推荐讲法：

> 每次桥接、每次确认、每次导出、每次发布交接，都会在本地落盘，有据可查。

### 亮点 5：人工确认发布边界非常清晰
推荐讲法：

> 这个版本明确不做自动直发。系统生成的是发布材料和交接页，最后一步必须是人工确认和人工发布。

---

## 5. 如何说明 OpenCode sandbox 多 Agent 团队

演示时建议用以下简化说法：

> 我们把 OpenCode 的多 Agent runtime 放在仓库内一个独立沙箱目录里运行，而不是直接污染主工作区。主应用通过 Python 编排层做桥接：把项目快照送进去，拿回 stdout、stderr、结果 JSON 和 manifest，再把桥接结果回写到项目状态里。

如果对方继续追问，可以补充：
- 沙箱目录：`sandbox/opencode-runtime/`
- 每次 bridge 都有 run-id
- 会落盘 snapshot / stdout / stderr / result / manifest
- 有验收清单来证明主 workspace 没被污染

---

## 6. 如何说明“人工确认发布、非自动直发”的边界

这是演示必须强调的一点，建议直接照着说：

> 当前版本只做到“生成发布素材 + 人工确认发布交接”。系统不会自动把内容发到抖音，也不会越过人工确认关口。我们保留的是一个清晰、安全、可审计的人机协作边界。

可以再补一句：

> 所以这里看到的是发布素材包、交接清单、本机打开目录/页面、复制文案等动作，而不是自动直发按钮。

## 7. 如果现场某一步失败，怎么兜底

### 情况 A：桥接驱动生成失败
兜底方式：
- 改用“生成全流程草案”继续演示

讲法建议：

> 当前 bridge 具备 fallback 机制。即使独立 sandbox 在现场环境中不可用，主流程仍可回退到本地 stub 生成，保证演示不中断。

### 情况 B：本机打开目录 / 打开交接页失败
兜底方式：
- 直接展示页面中记录的目标路径
- 打开文件管理器手动进入 `data/projects/<project-id>/exports/`

讲法建议：

> 本机调起动作本身也是受环境影响的，所以系统同时保留了 fallback 路径和 launch evidence，确保交接过程可追踪。

### 情况 C：不想逐项手动确认太耗时
兜底方式：
- 提前准备一个已生成、已确认的项目作为备用
- 现场只演示其中一两个阶段的确认动作，再切到完整状态项目展示后半程

### 情况 D：导出物太多，不知道看哪个
优先展示这几个：
- `export-bundle-vN.html`
- `manual-publish-handoff-vN.html`
- `export-evidence-vN.json`
- `runtime-bridge/acceptance-checklist.md`
